## Stack
- **Frontend**: React 18, TypeScript, Vite, Tailwind CSS, shadcn/ui, TanStack Query v5, Zustand, React Hook Form, Zod
- **Backend**: Node.js 20, Express, TypeScript, Knex.js (query builder only, no ORM)
- **DB**: PostgreSQL 16
- **Auth**: JWT access token (15m, in-memory) + Refresh Token (7d, httpOnly cookie)

## Architecture

### DB Schema Lifecycle
The initial schema is loaded by Docker via `db/init/001_schema.sql` (docker-entrypoint-initdb.d). **Do not edit this file for incremental changes.** All schema changes after the initial setup go into `backend/migrations/` as Knex migration files.

Migration files must be run manually with `npm run migrate` — they are not auto-applied. The knexfile is at `backend/src/config/knexfile.ts` and points to `../../migrations` (relative to its own location).

### Backend Layer Pattern
Every feature follows a strict 3-layer pattern:

```
routes/*.routes.ts      — Zod validation, auth/RBAC middleware, delegates to service
services/*.service.ts   — Business logic, throws AppError for domain errors
repositories/*.repository.ts — Raw Knex queries only, no business logic
```

- Throw `AppError(statusCode, message)` from services for expected errors (409 conflict, 404 not found, etc.). The global `errorHandler` in `middleware/errorHandler.ts` handles formatting.
- Use `requireRole('admin')` / `requireRole('admin', 'manager')` from `middleware/rbac.ts` to protect routes.
- Request body validation uses `validate({ body: schema })` middleware with a Zod schema.
- All routes in a router file use `router.use(authMiddleware)` at the top — public endpoints go in `auth.routes.ts` before any middleware.

### Frontend Layer Pattern
```
api/*.api.ts            — TanStack Query hooks (useQuery / useMutation) + type definitions
pages/*.tsx             — Page-level components, compose shared components
components/shared/      — Reusable: DataTable, FormField, ConfirmDialog, StatusBadge, etc.
components/ui/          — shadcn/ui primitives (Button, Input, Dialog, Select, Badge, etc.)
store/authStore.ts      — Zustand store for auth state (user, isLoading)
hooks/useAuth.ts        — Convenience wrapper over authStore + logout
```

- All API calls go through `api/client.ts` (Axios instance). It handles token attachment and auto-refresh on 401 using a queue pattern. Access token is stored in-memory only (never localStorage).
- When adding a new page: create the API hooks in `api/`, the page in `pages/`, register the route in `router.tsx`, and add a nav item to `components/layouts/Sidebar.tsx`.
- Sidebar nav items accept a `roles` array to control visibility by user role (`admin`, `manager`, `employee`).

### Auth Flow
1. Login → backend sets httpOnly refresh cookie + returns short-lived access token
2. Frontend stores access token in-memory (`api/client.ts` module variable)
3. On 401, Axios interceptor calls `/auth/refresh` (sends cookie automatically), retries original request
4. On refresh failure → clears token, redirects to `/login`
5. First login forces password change (`must_change_password` flag) — router redirects to `/change-password`

### RBAC
Three roles: `admin` > `manager` > `employee`. Backend enforces via `requireRole()` middleware. Frontend conditionally renders nav items and UI elements based on `user.role` from authStore.
